poxicle — Build, Install & Packaging

This page documents how poxicle is built and installed: the Meson engine and demos, the opt-in configurator and GObject binding, the separate CMake/ECM build for the KWin effect, the sanitizer setup, and the per-component install scripts.

1. The shape of the build

poxicle is one engine plus three integrations, and it uses two build systems because KDE mandates CMake/ECM for a KWin effect while everything else is Meson. There is no installed libpoxicle: the engine is small, dependency-free C, so each consumer compiles it straight from src/ and ships it inside itself.

Component Build system Command
Engine + demos Meson meson setup build && meson compile -C build
GTK4 configurator Meson (-Dconfigurator=true) adds poxicle-config
GObject binding + typelib Meson (-Dintrospection=true) adds libpoxicle-gobject + Poxicle-1.0.typelib
KWin effect + KCM CMake / ECM cd kwin && ./install.sh
GNOME extension + binding Meson + shell cd gnome && ./install.sh
Everything to /usr Meson + CMake ./install.sh

2. The Meson engine and demos

The top-level meson.build (project poxicle, C11, warning level 2, LGPL-3.0-or-later) defines the engine as two static libraries and the demos as two executables. Its dependencies are wayland-client, wayland-egl, egl, glesv2, and optional libm, plus wayland-scanner and wayland-protocols to generate xdg-shell glue for the demos.

Target Sources Links Installed
libpoxicle (static) poxicle.c, poxicle-gl.c glesv2, m no
libpoxicle-wl (static) poxicle-wl.c libpoxicle + wayland/egl no
poxicle-demo demo/main.c libpoxicle no
poxicle-host demo/host.c libpoxicle-wl + libpoxicle no

The core library carries only the simulation core and the GLES renderer — it needs nothing but a GL context, so it stays embeddable anywhere. The Wayland subsurface backend is a separate static lib that pulls in the windowing stack, kept out of the core on purpose. The build also declares poxicle / poxicle-wl as consumable Meson dependencies (meson.override_dependency), so an embedder like Chiguiro can pull poxicle in by name as a subproject. Only xdg-shell is generated from the system protocol description, and only the demos consume it; the libraries don’t.

3. The GObject-Introspection binding

Behind -Dintrospection=true (default off), the build adds the GObject binding: a shared library libpoxicle-gobject (from poxicle.c + pox-gobject.c, pulling in gobject-2.0), and a Poxicle-1.0 typelib generated by gnome.generate_gir — namespace Poxicle, version 1.0, identifier prefix Poxicle, symbol prefix poxicle. This is the only target that links GObject; the core stays clean.

Installed to a system prefix (default /usr), the binding lands where gjs and Python find it:

Artifact Path
libpoxicle-gobject.so /usr/lib/
Poxicle-1.0.gir /usr/share/gir-1.0/
Poxicle-1.0.typelib /usr/lib/girepository-1.0/

That last path is what makes import Pox from 'gi://Poxicle' resolve at runtime with no hard-coded path in the extension.

4. The KWin effect (CMake/ECM)

The KWin effect is a separate CMake project (kwin/CMakeLists.txt, C++20 / C11, AUTOMOC). It pins QT_DEFAULT_MAJOR_VERSION 6 before any find_package, then requires ECM, Qt6 (Core, Gui, Widgets, DBus, OpenGL, Qml, Quick), KF6 (CoreAddons, Config, KCMUtils, I18n), epoxy, glesv2, and KWin. It builds two plugins and one test tool:

Target Kind Installed to
poxicle_kwin the effect kwin/effects/plugins
kwin_poxicle_config the Configure KCM kwin/effects/configs
poxbridge-test a standalone stream producer (dev only) not installed

The effect compiles the engine straight from ../src — poxicle.c + poxicle-gl.c, no poxicle-wl — because KWin owns the surface and GL context. Two ECM-integration quirks are worth knowing: epoxy must come through ECM’s find_package(epoxy) (a pkg_check_modules with prefix epoxy collides with KWin’s transitive find), and glesv2 is brought in under a private PC_ prefix to avoid the same namespace clash.

5. Install scripts

Three scripts cover the three install shapes:

  • ./install.sh (top level) installs the engine, the configurator, and the KWin effect + KCM to /usr in one shot — Meson for poxicle-config, CMake for the effect. It always does a clean build (rm -rf build-install kwin/build-install first): the KWin effect bakes the compositor’s effect-API version into its plugin-factory IID, and an incremental rebuild after a KWin update can silently reinstall a plugin that names the old KWin, which the new KWin then refuses to load. Wiping is the only reliable cure. It forces -DKDE_INSTALL_PLUGINDIR=lib/qt6/plugins (the default can resolve to lib/qt/plugins, which KWin never scans), removes stale copies from the old path, and runs kbuildsycoca6.
  • kwin/install.sh installs only the effect, into a dedicated build-install dir so it doesn’t disturb the nested-KWin dev build, with the same plugin-dir flag and cache refresh.
  • gnome/install.sh installs the GI binding to the system prefix (so gjs finds gi://Poxicle) and symlinks the extension into ~/.local/share/gnome-shell/extensions/, then enables it.

The installed footprint is /usr/bin/poxicle-config, /usr/lib/qt6/plugins/kwin/effects/{plugins/poxicle_kwin.so, configs/kwin_poxicle_config.so}, and the binding artifacts above. Neither compositor effect hot-reloads recompiled code: applying new effect, engine, or extension code requires a real session restart (log out / log back in on Wayland), though palette and rule edits from the configurator apply live via the KWin reconfigure / the GNOME file monitor.

6. The demos

Two demos prove the two ways to host the engine, both built by default (-Ddemos=true):

  • poxicle-demo (demo/main.c) is the standalone path: a transparent Wayland/EGL window that owns its surface, EGL context, and frame loop, drives pox_engine_tick, and draws with pox_gl_render — no rendering code of its own.
  • poxicle-host (demo/host.c) is the subsurface backend path: a plain opaque wl_shm window standing in for “some app’s content,” with a poxicle overlay riding on a subsurface above it via pox_wl_new. It includes no EGL/GL headers at all — the context lives inside the backend — proving the embedding path is transparent and click-through and animates independently.

7. Sanitizers

The engine is meant to be developed under AddressSanitizer + UndefinedBehaviorSanitizer. Because consumers may link poxicle statically into a non-sanitized binary, the sanitizer coverage lives here, where the demos link the runtimes directly:


meson setup build-asan -Db_sanitize=address,undefined
meson compile -C build-asan

LSAN_OPTIONS=suppressions=lsan.supp \
ASAN_OPTIONS=abort_on_error=1 UBSAN_OPTIONS=halt_on_error=1:print_stacktrace=1 \
  build-asan/poxicle-host

lsan.supp filters out Mesa’s EGL/GL driver leaks — process-lifetime driver state allocated during context setup that appears under the backend’s egl_setup() but belongs entirely to the driver — so genuine poxicle leaks stand out. It suppresses the EGL/GLX loaders (libEGL_mesa, libGLX_mesa), the Gallium/DRI drivers (iris_dri, libgallium, swrast_dri, radeonsi, libglapi), and the GLVND dispatch (libglvnd, libGLdispatch).

8. Toolchain

The engine is verified against wayland-client 1.25, wayland-egl 18.1, EGL 1.5, GLES 3.2 (glesv2), wayland-scanner and Meson, on Mesa with an Intel Iris Plus GPU. The language is plain C, FFI-friendly for any consumer. The integrations layer on their own requirements: the KWin effect needs KWin/Plasma 6 (6.6.5+), Qt6, KF6, ECM and epoxy; the GObject binding needs gobject-2.0 + gobject-introspection; the configurator needs GTK4 + libadwaita-1; the GNOME extension needs GNOME Shell 50.

The wider integration story — how the Chiguiro terminal uses poxicle both as an in-app subsurface overlay and as a bridge producer streaming to the KWin effect — is captured in the repo’s docs/chiguiro-compositor-handoff.md.

9. Design rules

  • Two build systems, on purpose. KDE mandates CMake/ECM for a KWin effect; the engine, demos, configurator and binding are Meson. They meet only at the shared src/.
  • No shipped library. The engine is compiled into each consumer from src/, so there is no libpoxicle to version-skew against.
  • The core pulls nothing. Only the opt-in targets add Wayland, GObject, GTK, or KWin; the static core links just GLES and libm.
  • Clean-build the effect. The effect’s IID is tied to KWin’s API version, so installs wipe and rebuild to avoid shipping a plugin that names a stale compositor.
  • Sanitize where it links. ASAN/UBSAN coverage lives in poxicle’s own build because consumers may link it un-sanitized; lsan.supp keeps Mesa’s driver leaks out of the signal.
